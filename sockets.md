# 🌐 Сокеты: WebSockets и TCP-сокеты

## Введение

**WebSockets** — это технология, обеспечивающая постоянный двусторонний (полнодуплексный) канал связи между клиентом и сервером. В отличие от классического **HTTP**, где инициатором обмена всегда выступает клиент, WebSocket позволяет серверу отправлять данные клиенту в любой момент без предварительного запроса.

Это делает технологию идеальной для создания **real-time приложений**: чатов, онлайн-игр, живых дашбордов и систем совместной работы.

## ⚡ Преимущества WebSocket перед HTTP

| Критерий | HTTP | WebSocket |
| :--- | :--- | :--- |
| **Тип соединения** | Запрос-ответ (полудуплекс) | Постоянное соединение (дуплекс) |
| **Инициатор** | Только клиент | Клиент **и** Сервер |
| **Заголовки** | Большие (до 2 КБ) | Минимальные (после рукопожатия) |
| **Задержка** | Высокая (новый TCP-handshake) | Низкая (данные идут по уже открытому каналу) |
| **Нагрузка** | Высокая при частых опросах (Polling) | Низкая (только полезные данные) |

## 🚀 Ключевые сценарии использования

### 1. Интерактивные приложения в реальном времени
- **Чаты и мессенджеры**.  
  Пользователь А отправляет сообщение → сервер мгновенно проталкивает (push) его пользователю Б без задержек.
- **Онлайн-игры** (шахматы, стратегии, лобби).  
  Синхронизация состояния игры между всеми игроками за миллисекунды.
- **Совместное редактирование** (аналог Google Docs).  
  Десятки пользователей видят курсоры и правки друг друга в реальном времени (по символу).

### 2. Лайв-дашборды и мониторинг
- **Панели администратора**.  
  Обновление графиков нагрузки сервера, логов ошибок или курса криптовалют в реальном времени.
- **Биржевые стаканы и тикеры**.  
  Цена актива может меняться 100+ раз в секунду — постоянные HTTP-запросы были бы неэффективны.

### 3. Прогресс долгих операций
- **Тяжелые вычисления** (ML-модели, рендеринг видео).  
  Вместо того чтобы клиент опрашивал сервер каждые 5 секунд («Ну что там?»), сервер сам присылает обновления:  
  `"Обработано 15%... 87%... Готово!"`.

### 4. Микросервисы и внутренние стримы
- Передача потока событий между бэкенд-сервисами в реальном времени.  
  Альтернатива очередям (RabbitMQ, Kafka) в случаях, где **критична задержка**.

## 🛠 Принцип работы (кратко)

1. **Handshake (рукопожатие)** — клиент отправляет HTTP-запрос с заголовком `Upgrade: websocket`.
2. **Смена протокола** — сервер отвечает кодом `101 Switching Protocols`.
3. **Обмен данными** — после этого соединение переключается на WebSocket-протокол, и стороны могут отправлять сообщения в любой момент.

---

## 💡 Заключение

WebSockets — это стандарт для всего, что требует **мгновенной доставки данных** и минимизации накладных расходов. Используйте их, если ваше приложение не может позволить себе задержки HTTP-опроса (Polling) или Long Polling.
