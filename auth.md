# Выбор способа авторизации: руководство

## Введение

Существует множество способов реализации авторизации, и каждый из них имеет свои плюсы и минусы. Выбор подхода зависит от архитектуры приложения, требований к безопасности и масштабируемости. Ниже приведён обзор основных методов, а также обоснование выбора JWT с хранением в cookies для данного проекта.

---

## Основные виды авторизации

| Метод | Описание | Типичное применение |
|-------|----------|----------------------|
| **JWT (JSON Web Token)** | Зашифрованный токен, содержащий информацию о пользователе и срок действия. Передаётся при каждом запросе. | REST API, микросервисы, SPA. |
| **OAuth2** | Протокол авторизации через сторонние сервисы (Google, Яндекс и др.). | Большие системы, микросервисная архитектура. |
| **Session-based** | Сервер хранит сессию, клиент получает session ID в cookie. | Традиционные веб-приложения. |
| **Basic Authentication** | Логин и пароль передаются в заголовке (Base64). | Тесты, внутренние сервисы (устаревает). |
| **API Key** | Уникальный ключ передаётся в каждом запросе. | Авторизация приложений / автоматизированных клиентов. |
| **OpenID Connect** | Надстройка над OAuth2 для идентификации пользователя и SSO. | Единый вход (Single Sign-On). |

## Выбранный подход: JWT с хранением в cookies

В рамках данного проекта принято решение использовать **JWT**, хранящийся в **HttpOnly cookie**.

### Преимущества JWT
- **Stateless** — сервер не хранит сессии, что упрощает масштабирование и повышает производительность.
- **Гибкость** — токен может содержать роли, права доступа и прочие данные, что снижает количество запросов к БД.
- **Производительность** — проверка подписи происходит локально, без обращения к хранилищу сессий.
- **Безопасность** — при хранении в HttpOnly cookie токен недоступен из JavaScript, что защищает от XSS-атак.

### Недостатки JWT
- **Компрометация токена** — при утечке злоумышленник может использовать его до истечения срока.
- **Размер** — токен может быть крупным, увеличивая сетевой трафик.
- **Управление сроком действия** — требуется механизм обновления (refresh token).
- **Отсутствие отзыва** — стандартные JWT нельзя отозвать до истечения срока.

### Почему cookies, а не заголовки?

| Способ передачи | Особенности |
|----------------|-------------|
| **HTTP-заголовок (Bearer)** | Популярен для внешних API. Токен доступен из JS, уязвим к XSS. |
| **Cookie (HttpOnly)** | Браузер автоматически отправляет cookie. Токен недоступен из JS, защищён от XSS. |
| **Body / Query** | Наименее безопасный вариант (используется крайне редко). |

**Итог:** хранение в cookie с атрибутами `HttpOnly`, `Secure`, `SameSite` повышает безопасность, а указание `max-age` или `expires` позволяет браузеру автоматически удалять токен по истечении срока.

## Алгоритм реализации (по шагам)

1. **Установка библиотеки**  
   ```bash
   poetry add pyjwt
   ```

2. **Генератор Redis-сессий**
   - Написать класс для хранения сессий в Redis (опционально, для поддержки механизма отзыва или refresh-токенов).

3. **Pydantic-схемы**
   - Создать схемы для приёма и отдачи данных (логин/пароль, ответ с токеном и т.д.).

4. **Маршрут авторизации**
   - Добавить эндпоинт (например, /auth/login), принимающий учётные данные.

5. **Сервис авторизации**

   - Получение пользователя из БД.
   - Проверка пароля.
   - Формирование JWT (с нужными полями и сроком действия).
   - Установка cookie с токеном.
   - Возврат ответа клиенту.
