# Что длеать если проблема с серваком

##  пользователю необходимо просматривать списки процессов и останавливать процессы, а также просматривать имеющиеся и потребляемые ресурсы сервера в системе.

Ниже приведены несколько примеров наиболее часто используемых команд:

# руководство (справочная информация)
```bash
man <utility> # просмотр справочной информации по утилите
<utility> --help # просмотр справочной информации по утилите
```
# процессы
```bash
top
sudo ps aux
sudo kill -9 <pid> # убийство процесса по PID
sudo killall -s 9 <name> # убийство всех процессов по имени
```
# утилиты для мониторинга использования ресурсов
```bash
htop # использование ресурсов по процессам (может потребоваться установка пакета htop)
nmon # использование процессора по ядрам, памяти, дисков и др. (может потребоваться установка пакета nmon)
iostat # использование процессора в среднем по ядрам и чтение/запись по дискам
```
# ЦПУ
```bash
lscpu # общая информация
cat /proc/cpuinfo # подробная информация
```
# ОЗУ
```bash
cat /proc/meminfo # общая информация и потребление
free -h # удобное представление на основе данных из файла meminfo
ps aux --sort -rss # использование памяти по процессам
```
# диски
```bash
lsblk # общая информация
df -h # просмотр занятного места по разделам
du -сh <dir> # просмотр занятого места в каталоге
```
# сеть
```bash
/sys/class/net/<interface>/speed # просмотр максимальной скорости интерфейса
```
# Алгоритм проверки:

## Что делать если проблема с серваком

## Базовый функционал для администратора
Пользователю необходимо просматривать списки процессов, останавливать их, а также анализировать нагрузку на ресурсы сервера.

### Основные команды
* **Справочная информация:** `man <utility>`, `<utility> --help`
* **Процессы:** `top`, `sudo ps aux`, `sudo kill -9 <pid>`, `sudo killall -s 9 <name>`
* **Мониторинг:** `htop`, `nmon`, `iostat`
* **ЦПУ:** `lscpu`, `cat /proc/cpuinfo`
* **ОЗУ:** `cat /proc/meminfo`, `free -h`, `ps aux --sort -rss`
* **Диски:** `lsblk`, `df -h`, `du -ch <dir>`
* **Сеть:** `cat /sys/class/net/<interface>/speed`

## Алгоритмы диагностики (Troubleshooting)

### 1. Сервер не пингуется
1.  `ping <IP>` — проверка связи.
2.  `ip a` — проверка наличия IP и статуса сетевых интерфейсов.
3.  `ip route` — проверка шлюза по умолчанию.
4.  `sudo iptables -L -n` / `sudo ufw status` — проверка правил фаервола.

### 2. Сервер пингуется, но приложение выдает 500
1.  `systemctl status <service_name>` / `docker ps` — проверка статуса сервиса.
2.  `journalctl -u <service_name> -f` / `docker logs --tail 100 <container>` — изучение логов приложения.
3.  `htop` — проверка перегрузки CPU/RAM.
4.  `ss -tulpn | grep <port>` — проверка, слушает ли приложение нужный порт.

### 3. Серверу не хватает памяти (OOM)
1.  `free -h` — проверка доступной памяти.
2.  `ps aux --sort=-%mem | head -n 10` — поиск процесса-пожирателя памяти.
3.  `dmesg -T | grep -i oom` — поиск событий OOM-killer в ядре.
4.  `swapon --show` — проверка состояния Swap.

### 4. Проблема с докером (проект не собирается)
1.  `systemctl status docker` — проверка состояния демона.
2.  `df -h` — проверка наличия свободного места на разделах (особенно `/var/lib/docker`).
3.  `docker system prune` — очистка неиспользуемых контейнеров/сетей.
4.  `docker build --progress=plain .` — детальный вывод процесса сборки.

<summary><b>Быстрый чек-лист (команда-спасатель)</b></summary>

Если сервер «тормозит» или завис:
1. `top` — смотрим Load Average и %CPU.
2. `free -m` — смотрим ОЗУ.
3. `df -h` — смотрим свободное место.
4. `dmesg | tail -n 20` — последние критические события ядра.
5. `journalctl -f` — просмотр системных логов в реальном времени.
</details>

#### 5. Кейс от пересборок засрался докер

`docker builder prune -a` - справления ошибок всегда полезно выполнить глубокую очистку, если накопилось много «битых» слоев

#### 6. Перезапустить после правок 

1. Просто повторить -	`docker build -t <имя> .`
2. Изменения в файлах/пакетах не подтянулись	`docker build --no-cache -t <имя> .`
3. Сборка постоянно прерывается на «битых» слоях	`docker builder prune -a и затем docker build --no-cache ...`
